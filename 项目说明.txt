舆情智能标注与分流决策系统
================================

版本: v1.7.0
日期: 2026-05-16
代码规模: 6,690行 Python (engine + UI + tests)
测试覆盖: 69项，全平台零回归
GitHub: https://github.com/evanhan1995/opinion-annotation-demo


一、项目是什么
--------------

一个基于大语言模型（LLM）与 Wiki 知识库驱动的舆情智能标注系统。
用户输入社交媒体链接，系统自动抓取内容 + 评论区 → AI 完成严重度
评级（P0-P3）、分流建议、情感分析、风险标签、评论区红绿灯 →
自动存入知识库 → 知识库驱动 AI 标注准确率持续提升。

系统设计理念源自 Andrej Karpathy 的 LLM Wiki 方法论：知识库不是
静态文档，而是由 AI 全权维护、随案例增长的"活文档"。每一个新标
注案例都会自动回灌到知识库中，校准未来的标注决策边界。


二、核心能力
------------

1. 多平台舆情抓取
   支持 5 个平台：YouTube、小红书、抖音、Reddit、X (Twitter)
   - YouTube: yt-dlp 抓取，含标题/描述/播放量/评论，可选字幕抓取
   - 小红书: XHS-Downloader 无 Cookie 元数据抓取 + xhshow API 评论抓取
   - 抖音: TikTokDownloader 抓取，含 147 字段元数据 + 评论
   - Reddit / X: Playwright 浏览器自动化抓取

2. AI 自动标注
   - 严重度评级: P0(紧急) / P1(立即处理) / P2(持续观察) / P3(可忽略)
   - 分流建议: 立即处理 / 持续观察 / 可忽略 / 正面可利用
   - 情感分析: 多维度情感标签
   - 风险标签: 安全 / 质量 / 客服 / 竞品攻击 / KOL负面 / 大规模传播 等
   - 评论区红绿灯: 红(负面主流) / 黄(混合/争议) / 绿(正面/可控)
   - 舆情分类: 商品问题 / 商品侵权 / 售后问题 / 数据泄露 / 软件问题 等
   - 流式输出: 标注结果逐字生成，实时可见
   - 近似舆情匹配: 自动匹配知识库中最相似的已有案例

3. 知识库引擎
   - 自动 Ingest: 每次标注完成后自动生成 Wiki 案例页面
   - 活文档: 案例驱动标注规范持续校准
   - 三维索引: 按严重度 / 分流建议 / 平台三维交叉检索
   - 跨条目关联: 自动检测跨平台事件的碎片化舆情并聚合
   - 作者库: 跨平台作者/账号聚合追踪
   - 巡检监控: 指定 URL 列表定时检查新增变化
   - P0/P1 告警: 严重案例自动醒目标识

4. 人工纠偏
   - 支持人工修改 AI 标注结果
   - 差异对比: 自动 diff 人工 vs AI，高亮显著变更
   - 纠偏回灌: 人工修正结果作为校准案例存入知识库

5. 智能问答（扫地僧）
   - 基于知识库的自然语言问答
   - 支持跨平台关联查询
   - 回答附带引用来源链接

6. Web 操作界面
   - Streamlit Web UI，5 个 Tab
   - Tab1 手工录入: 手动粘贴舆情内容进行标注
   - Tab2 URL 抓取: 粘贴链接自动抓取+标注，支持批量模式
   - Tab3 知识库浏览器: 浏览/搜索所有 Wiki 页面，密码保护
   - Tab4 扫地僧: 自然语言问答知识库
   - Tab5 操作演示: 引导式 Demo
   - 部署: Streamlit Cloud 一键部署


三、解决什么问题
----------------

1. 舆情研判效率问题
   传统人工舆情研判需要分析师逐条阅读、评级、分流。本系统将
   单条舆情的处理时间从 ~5分钟 压缩到 ~3秒，加速约 100 倍。

2. 标注一致性难题
   不同分析师对同一舆情的评级可能不一致。系统通过知识库案例驱
   动的标注规范，实现可追溯、可复现的标准化标注。人工纠偏机制
   进一步保证标注质量，历史纠偏率约 67%。

3. 跨平台碎片化
   同一舆情事件可能分散在 YouTube、小红书、抖音等多个平台。
   系统自动检测跨条目关联，聚合碎片化舆情为完整事件画像。

4. 知识沉淀与传承
   资深分析师的判断经验往往难以系统化传递。系统的知识库引擎
   将每一次标注转化为结构化案例，形成可检索、可引用的组织记忆。

5. 舆情监控闭环
   从"发现问题"到"分析问题"到"给出建议"到"归档学习"，
   形成完整的信息闭环。不再是"看过就忘"的一次性操作。

6. 批量处理能力
   支持批量 URL 导入，一次性处理多条舆情，适合定期巡检和
   集中排查场景。

7. 实时评论区分析
   不仅分析帖子本身，还抓取并分析评论区情绪走向，识别
   "帖子无害但评论区翻车"的隐性风险。

8. 多平台覆盖
   覆盖国内外主流社交媒体平台（YouTube、小红书、抖音、
   Reddit、X），无需为每个平台配置单独的工具。


四、适用场景
------------

- 出海品牌舆情监控: 监控海外主流平台（YouTube、Reddit、X）
  及国内出海平台（TikTok、小红书）的品牌舆情
- 电商平台客服质检: 批量分析用户投诉帖，快速判定严重度
- 竞品舆情分析: 系统性追踪竞品的负面/正面舆情
- 突发事件响应: 快速评估突发舆情事件的严重程度和扩散风险
- 舆情分析团队日常: 替代人工初筛，人工仅处理边界案例和纠偏


五、技术架构
------------

架构模型（信息论视角）:

  URL → [Scraper 5平台] → raw/cases/
    → [Annotate LLM标注] → outputs/
    → [Ingestor 知识库引擎] → wiki/cases/
    → [Agent 扫地僧] ← 知识库检索

核心原则:
  - raw/ 永不修改（原始数据只追加）
  - wiki/ 由 AI 全权维护（人类通过投放案例间接影响）
  - 每次操作有日志（wiki/log.md append-only）
  - 规则来自案例（不凭空调整标注规范）
  - 信息闭环（任何自动生成的信息可被未来系统消费）


六、环境要求
------------

- Python >= 3.12
- Streamlit (Web UI)
- Playwright (Reddit / X 抓取)
- yt-dlp (YouTube 抓取)
- XHS-Downloader (小红书元数据抓取，可选)
- TikTokDownloader (抖音抓取，需 Cookie)
- DeepSeek / OpenAI API (LLM 标注)
- 详见 requirements.txt
