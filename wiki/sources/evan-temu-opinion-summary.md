# Evan 在拼多多/TEMU 的舆情体系工作总结

## 核心观点

TEMU 舆情体系是一个**三层人机协作系统**：监测组（信息采集）→ 指挥部（分析分流）→ 建联侧（执行闭环）。每层有独立的 KPI 和操作规范，层级间通过系统进行信息传递。

## 系统架构

```
监测组（8×3 24h）  →  指挥部（打标+分流）  →  建联侧（处置闭环）
    │                      │                      │
  信息采集              分析+决策              执行+反馈
```

## 三层职责提取

### 监测组
- **工具组合**：Meltwater（国外）+ 新浪舆情通（国内）+ 人工搜索补齐
- **覆盖范围**：新闻、X、YouTube、Instagram、TikTok、其他社媒
- **录入字段**：链接、国家、发布者、影响力（播放量/粉丝量/赞评数）
- **附加职能**：正面 UGC 内容挖掘 → 转 PR 侧二次筛选
- **KPI 核心**：漏检率（最重要）、发布到维护时间、被忽略率、维护质量

### 指挥部（核心分析节点）
- **打标任务**：问题类型分类 + 严重度评级
- **三分流判断**：ignoring（忽略）/ processing（处理）/ observing（观察）
- **高危响应**：立即拉群 → 盯进展 → 内部核查 → 结案归档
- **归档内容**：发生时间、内容摘要、发现人/时间、当班人员、处置时间线、最终结果
- **日常输出**：每日舆情日报 + 建联日报 + 月度大月报
- **KPI 核心**：高危响应速度（主要）、高危漏检率（重要）、汇报能力
- **月报维度**：P0/P1 统计、处理成功率、红绿灯达人标注（红灯=负面高危，绿灯=正面可合作）

### 建联侧
- **终审评估**：对指挥部任务做最终判断，可申请忽略
- **处置执行**：法务/PR 手段进行建联处置
- **闭环反馈**：处置结果（删除/404/已澄清/放弃）反馈指挥部

## 关键引用

> "监测组成员认为负面内容均维护入系统"

> "对于高危舆情，除正常维护外，需立即反馈指挥部"

> "指挥部负责对于监测组维护内容进行打标（具体问题类型，评级）、分析、判断，是否确实需要处理（ignoring/processing/observing）"

## 个人评注

TEMU 体系最值得 AI 化的环节是指挥部的**打标+分流判断**。这是整个系统的信息压缩节点——所有采集到的内容在此被分类、评级、分流。这个节点的质量直接决定了漏检率和误报率。从信息论角度看，这是系统中**熵压缩最剧烈**的一步：从非结构化文本到离散决策标签。

三层架构的另一个启示：**人始终在回路中**。AI 可以替代监测组 80% 和指挥部 70% 的工作，但建联侧的终审评估（法务风险、商业判断）几乎无法替代。系统设计时应尊重这个边界。
